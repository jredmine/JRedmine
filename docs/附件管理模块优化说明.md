# 附件管理模块优化说明

## 优化内容

根据项目规范，对附件管理模块进行了以下三项优化：

### 1. 移除 XML 配置，直接使用 MyBatis-Plus

**优化前：**
- 使用 `AttachmentMapper.xml` 配置文件
- Mapper接口中定义方法，XML中实现SQL
- 在Mapper接口上使用 `@Update` 注解写SQL

**优化后：**
- 删除 `AttachmentMapper.xml` 文件
- Mapper 接口只继承 `BaseMapper<Attachment>`，不需要额外方法
- 直接使用 MyBatis-Plus 提供的 `updateById` 方法
- 与 IssueMapper、TimeEntryMapper 等其他模块保持一致

```java
@Mapper
public interface AttachmentMapper extends BaseMapper<Attachment> {
    // 不需要任何额外方法，直接使用 MyBatis-Plus 提供的方法
}
```

**Service 中使用：**
```java
// 增加下载次数
attachment.setDownloads(attachment.getDownloads() + 1);
attachmentMapper.updateById(attachment);
```

**优点：**
- 代码更简洁，Mapper 接口非常干净
- 充分利用 MyBatis-Plus 的能力，不需要手写SQL
- 与项目其他模块风格保持一致
- 更易于维护和理解

### 2. 封装上传接口参数为 RequestDTO

**优化前：**
```java
public ApiResponse<AttachmentResponseDTO> uploadAttachment(
    @RequestParam("file") MultipartFile file,
    @RequestParam(value = "containerType", required = false) String containerType,
    @RequestParam(value = "containerId", required = false) Long containerId,
    @RequestParam(value = "description", required = false) String description)
```

**优化后：**
```java
public ApiResponse<AttachmentResponseDTO> uploadAttachment(
    @RequestParam("file") MultipartFile file,
    @Valid AttachmentUploadRequestDTO request)
```

**优点：**
- 参数更集中，便于维护
- 支持参数验证（@Valid）
- 便于扩展，添加新参数不需要修改方法签名
- 与 IssueController 等其他模块保持一致

### 3. 删除冗余的容器查询接口

**优化前：**
- 有两个接口可以查询容器的附件：
  1. `GET /api/attachments?containerType=Issue&containerId=1` （通用查询接口）
  2. `GET /api/attachments/container/Issue/1` （专用容器查询接口）

**优化后：**
- 只保留通用查询接口 `GET /api/attachments`
- 通过 `containerType` 和 `containerId` 参数实现容器查询
- 删除了 Controller 中的 `getAttachmentsByContainer` 方法
- 删除了 Service 中的 `getAttachmentsByContainer` 方法

**优点：**
- 减少API数量，避免功能重复
- 接口更统一，使用方式更一致
- 降低维护成本

## 使用示例对比

### 查询容器的附件

**优化前（两种方式）：**
```bash
# 方式1: 使用通用查询接口
curl -X GET "http://localhost:8088/api/attachments?containerType=Issue&containerId=1" \
  -H "Authorization: Bearer YOUR_TOKEN"

# 方式2: 使用专用容器查询接口（已删除）
curl -X GET "http://localhost:8088/api/attachments/container/Issue/1" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

**优化后（统一方式）：**
```bash
# 统一使用通用查询接口
curl -X GET "http://localhost:8088/api/attachments?containerType=Issue&containerId=1" \
  -H "Authorization: Bearer YOUR_TOKEN"
```

## 优化后的接口列表

| ------------ | ------ | ------------------------------ | ---------------------------------- |
| 上传附件     | POST   | /api/attachments               | 上传文件并创建附件记录             |

**接口数量：** 从原来的 7 个减少到 6 个

## 总结

经过这三项优化，附件管理模块的代码更加规范，与项目整体风格保持一致：

1. **代码风格统一**：与 Issue、TimeEntry 等模块保持一致的实现方式
2. **接口更简洁**：减少冗余接口，降低维护成本
3. **参数更规范**：使用 RequestDTO 封装参数，便于扩展和维护
4. **SQL更集中**：使用注解方式，代码和SQL在同一处，便于查看和修改

这些优化提高了代码的可维护性和可读性，是遵循项目最佳实践的重要改进。
