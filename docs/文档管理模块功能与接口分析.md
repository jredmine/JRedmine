# 文档管理模块功能与接口分析

基于现有数据库（`documents`、`enumerations` 文档分类）、权限枚举及附件多态设计，整理文档管理模块的功能与 REST 接口方案。

---

## 一、现有基础

| 项目 | 状态 | 说明 |
|------|------|------|
| 表结构 | ✅ | `documents`（id, project_id, category_id, title, description, created_on） |
| 文档分类 | ✅ | `enumerations` 表，type='DocumentCategory'，支持 project_id 区分全局/项目分类 |
| 权限 | ✅ | view_documents、add_documents、edit_documents、delete_documents |
| 项目模块 | ✅ | ProjectModule.DOCUMENTS，项目可启用/禁用文档模块 |
| 附件 | ✅ | 附件已支持 containerType=Document、containerId=文档ID |
| 实体/Mapper | ❌ | 需新增 Document 实体、DocumentMapper、文档分类查询 |

**说明**：文档在库中为“元数据”（标题、描述、分类）；实际文件通过附件关联（container_type=Document, container_id=文档 id）。创建文档 = 写一条 documents 记录，上传文件 = 走现有附件接口并指定 Document 容器。

---

## 二、功能概览

### 1. 文档分类（DocumentCategory）

- **列表**：查询某项目可用的文档分类（enumerations 中 type='DocumentCategory' 且 (project_id IS NULL OR project_id=当前项目)，按 position/name 排序）。
- **可选**：项目管理员可增删改项目级文档分类（写 enumerations，type='DocumentCategory', project_id=项目ID）。若不做独立接口，可与“枚举/设置”统一或后续迭代。

### 2. 文档 CRUD

- **创建**：在项目下新增一条 `documents`（title 必填，description、category_id 可选）。category_id=0 表示未分类。
- **列表**：分页列表，支持按 project_id、category_id、关键词（title/description 模糊）筛选，按 created_on 倒序。
- **详情**：按 documentId 返回文档信息；可选同时返回该文档下的附件列表（调用现有附件列表接口或内部查 container_type=Document, container_id=id）。
- **更新**：更新 title、description、category_id。
- **删除**：删除 documents 记录；是否级联删除附件由策略决定（建议：仅删文档记录，附件可保留或同步删除，需与现有附件删除策略一致）。

### 3. 权限与可见性

- **项目模块**：仅当项目启用 `documents` 模块时，文档接口对该项目可用。
- **权限**：列表/详情 view_documents；创建 add_documents；更新 edit_documents；删除 delete_documents；分类管理（若做）可用 edit_documents 或单独权限。

### 4. 与附件的配合

- 创建文档后，前端上传文件时调用现有**附件上传**接口，传 `containerType=Document`、`containerId=文档ID`。
- 文档详情或列表可返回附件数量或附件列表（通过现有附件查询接口按 container 过滤即可）。

---

## 三、建议的 REST 接口清单

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | `/api/projects/{projectId}/document-categories` | 文档分类列表（供下拉/筛选） | view_documents |
| GET | `/api/projects/{projectId}/documents` | 文档分页列表（categoryId、keyword、current、size） | view_documents |
| POST | `/api/projects/{projectId}/documents` | 创建文档（title、description、categoryId） | add_documents |
| GET | `/api/projects/{projectId}/documents/{documentId}` | 文档详情（可选带附件列表或附件数） | view_documents |
| PUT | `/api/projects/{projectId}/documents/{documentId}` | 更新文档 | edit_documents |
| DELETE | `/api/projects/{projectId}/documents/{documentId}` | 删除文档（是否删附件见实现策略） | delete_documents |

**附件**：上传/下载/删除文档附件均使用现有附件接口，参数 `containerType=Document`、`containerId=文档ID`，无需新增接口。

## 四、数据模型与 DTO 建议

### 1. 实体

- **Document**：id, projectId, categoryId, title, description, createdOn。与表 `documents` 一一对应。

### 2. 请求 DTO

- **DocumentCreateRequestDTO**：title（必填）, description, categoryId（可选，0 表示未分类）。
- **DocumentUpdateRequestDTO**：title, description, categoryId（均可选，只更新非空字段）。
- **DocumentListRequestDTO**（或直接用参数）：projectId（路径）, categoryId, keyword, current, size。

### 3. 响应 DTO

- **DocumentListItemResponseDTO**：id, projectId, title, description, categoryId, categoryName, createdOn；可选 attachmentCount。
- **DocumentDetailResponseDTO**：在列表项基础上增加附件列表或 attachmentIds（由现有附件接口或内部查询得到）。

### 4. 文档分类

- **DocumentCategoryResponseDTO**：id, name, position, projectId（null 表示全局）。  
  或直接复用 Enumeration 的现有结构，仅约定 type=DocumentCategory。

## 五、开发顺序建议

1. **Phase 1**：Document 实体 + DocumentMapper；项目启用 documents 模块校验；文档 CRUD（创建/列表/详情/更新/删除）；文档分类列表（只读，从 enumerations 查）。
2. **Phase 2**：列表返回附件数量或附件摘要；删除文档时级联删除附件（或明确“只删文档不删附件”并在文档中说明）。
3. **Phase 3**（可选）：项目级文档分类的增删改（若产品需要）。

---

## 六、小结

| 类别 | 内容 |
|------|------|
| 表 | documents（主表）、enumerations（文档分类） |
| 权限 | view_documents、add_documents、edit_documents、delete_documents |
| 核心接口 | 分类列表、文档列表、创建、详情、更新、删除 |
| 附件 | 沿用现有附件 API，containerType=Document、containerId=文档 id |
| 与 Wiki 差异 | 文档为“元数据+附件”，无正文版本历史；分类来自 enumerations，结构更简单 |

按上述清单可实现与现有项目、权限、附件体系一致的文档管理模块；实现时需补 Document 实体、Mapper 及 DocumentService、DocumentController。
