# 角色与权限管理模块接口清单

> 本文档列出 JRedmine 角色与权限管理模块需要实现的所有接口，按照优先级分类。

## ⚠️ 重要说明

### 1. 用户账号本身没有角色区分

**用户表（users）**：
- 只有 `admin` 字段，用于区分是否是**系统管理员**
- 用户本身**没有全局角色**
- 系统管理员（admin = true）拥有所有权限，不受项目限制

### 2. 角色是项目级别的，不是用户级别的

**权限体系结构**：
```
users (用户)
  ↓
members (项目成员: user_id + project_id)
  ↓
member_roles (成员角色: member_id + role_id)
  ↓
roles (角色: 包含权限列表)
```

**关键理解**：
- 用户必须先加入项目（成为 member），才能有角色
- 同一个用户在不同项目中可以有不同的角色
- 角色是项目级别的，不是全局的

### 3. 权限是固定的枚举值

**权限特点**：
- 权限列表是**预定义的枚举值**（如 `view_issues`, `add_issues` 等）
- 不需要复杂的动态权限系统
- 权限可以分配给不同的角色
- 角色可以动态 CRUD，但权限列表是固定的

### 4. 必须先实现项目模块

**依赖关系**：
- `member_roles` 依赖 `members` 表
- `members` 表依赖 `projects` 表
- 因此，**必须先实现项目模块**，才能实现成员角色分配功能

**实施顺序建议**：
1. ✅ 用户管理模块（已完成）
2. ⏳ **项目管理模块**（必须先做）
3. ⏳ 角色与权限模块（依赖项目模块）
4. ⏳ 任务管理模块（依赖项目和权限）

## 📊 数据库表结构分析

### 1. roles 表（角色表）

```sql
CREATE TABLE `roles` (
  `id` int NOT NULL AUTO_INCREMENT,
  `name` varchar(255) NOT NULL DEFAULT '',           -- 角色名称
  `position` int DEFAULT NULL,                       -- 排序位置
  `assignable` tinyint(1) DEFAULT '1',                -- 是否可分配
  `builtin` int NOT NULL DEFAULT '0',                 -- 是否内置角色（0=自定义，1-5=内置）
  `permissions` text,                                 -- 权限列表（JSON格式）
  `issues_visibility` varchar(30) DEFAULT 'default',  -- 任务可见性
  `users_visibility` varchar(30) DEFAULT 'members_of_visible_projects', -- 用户可见性
  `time_entries_visibility` varchar(30) DEFAULT 'all', -- 工时可见性
  `all_roles_managed` tinyint(1) DEFAULT '1',          -- 是否管理所有角色
  `settings` text,                                    -- 设置（JSON格式）
  `default_time_entry_activity_id` int DEFAULT NULL, -- 默认工时活动ID
  PRIMARY KEY (`id`)
)
```

**关键字段说明：**
- `builtin`: 内置角色（0=自定义，1=非成员，2=匿名，3=管理员等）
- `permissions`: 权限列表，存储为文本（可能是序列化的数组）
- `issues_visibility`: 任务可见性（`all`, `default`, `own`）
- `users_visibility`: 用户可见性（`all`, `members_of_visible_projects`）

### 2. member_roles 表（成员角色关联表）

```sql
CREATE TABLE `member_roles` (
  `id` int NOT NULL AUTO_INCREMENT,
  `member_id` int NOT NULL,                           -- 项目成员ID
  `role_id` int NOT NULL,                             -- 角色ID
  `inherited_from` int DEFAULT NULL,                   -- 继承自（子项目继承父项目角色）
  PRIMARY KEY (`id`)
)
```

**说明：**
- 一个项目成员可以有多个角色
- 支持从父项目继承角色

### 3. workflows 表（工作流表）

```sql
CREATE TABLE `workflows` (
  `id` int NOT NULL AUTO_INCREMENT,
  `tracker_id` int NOT NULL DEFAULT '0',              -- 跟踪器ID
  `old_status_id` int NOT NULL DEFAULT '0',           -- 原状态ID
  `new_status_id` int NOT NULL DEFAULT '0',           -- 新状态ID
  `role_id` int NOT NULL DEFAULT '0',                 -- 角色ID
  `assignee` tinyint(1) DEFAULT '0',                  -- 是否仅分配给指定人
  `author` tinyint(1) DEFAULT '0',                    -- 是否仅创建人
  `type` varchar(30) DEFAULT NULL,                    -- 类型
  `field_name` varchar(30) DEFAULT NULL,               -- 字段名
  `rule` varchar(30) DEFAULT NULL,                    -- 规则
  PRIMARY KEY (`id`)
)
```

**说明：**
- 定义任务状态转换规则
- 控制哪些角色可以执行哪些状态转换

### 4. roles_managed_roles 表（角色管理关系表）

```sql
CREATE TABLE `roles_managed_roles` (
  `role_id` int NOT NULL,                             -- 角色ID
  `managed_role_id` int NOT NULL,                     -- 被管理的角色ID
  UNIQUE KEY (`role_id`, `managed_role_id`)
)
```

**说明：**
- 定义角色之间的管理关系
- 某些角色可以管理其他角色

## 📊 Redmine 内置角色

Redmine 有以下内置角色（builtin 值）：

- `0` - 自定义角色
- `1` - 非成员（Non member）
- `2` - 匿名用户（Anonymous）
- `3` - 管理员（Administrator）

**注意**: 内置角色不能删除，且某些字段不能修改。

## 📌 注意事项

1. **用户账号没有角色**: 用户表只有 `admin` 字段，用户本身没有全局角色
2. **角色是项目级别的**: 角色通过 `members` → `member_roles` → `roles` 关联，是项目级别的权限
3. **系统管理员**: `admin = true` 的用户拥有所有权限，不受项目限制
4. **权限是固定枚举**: 权限列表是预定义的，不需要动态创建权限
5. **内置角色保护**: 内置角色（builtin > 0）不能删除，且某些字段不能修改
6. **权限验证**: 所有接口都需要进行权限验证，确保只有有权限的用户才能操作
7. **数据一致性**: 删除角色前需要检查是否有项目成员使用此角色
8. **权限格式**: `permissions` 字段存储格式需要确定（JSON数组或序列化字符串）
9. **可见性控制**: `issues_visibility`、`users_visibility`、`time_entries_visibility` 需要实现相应的过滤逻辑
10. **依赖关系**: 成员角色分配功能必须先实现项目模块
